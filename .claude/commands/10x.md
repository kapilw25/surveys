# /10x this — what's the path to 10× better?

Target: $ARGUMENTS (if blank, take the most recent solution, system, design, or approach from the conversation).

## What this is NOT

- Not `/brutal` (backward critique) — this is forward-looking.
- Not `/missing` (gap enumeration) — this is about IMPROVING what's working.
- Not generic "ideas" — every move must name a CONSTRAINT being relaxed.

## Protocol

### Step 1 — One-line state of the current solution
"Currently the solution does X with cost ≈ Y, speed ≈ Z, reliability ≈ W, scale ≈ S, value-per-unit ≈ V."
If any number is unknown, write `?` — don't fabricate.

### Step 2 — For each 10× axis, name the binding constraint
A solution rarely 10× across every axis. Pick the axes where 10× is even physically possible for THIS artifact:

```text
┌─────────────────┬──────────────────────────────────────────────────────────────────────┐
│ 10× axis         │ Question to ask                                                       │
├─────────────────┼──────────────────────────────────────────────────────────────────────┤
│ Speed (latency)  │ What forces this to be slow? What's the floor if that goes away?     │
│ Cost ($)         │ Where does each $ go? What's the cheapest substitute per unit?       │
│ Scale (volume)   │ What breaks at 10× volume? What change keeps it linear?              │
│ Reliability      │ What's the dominant failure mode? What removes it?                   │
│ Simplicity       │ What's the most LoC / config / steps? What collapses it?             │
│ Robustness       │ What input class breaks this? What makes it gracefully degrade?      │
│ Value-per-unit   │ What does each user/run actually get? What 10× the per-unit gain?    │
└─────────────────┴──────────────────────────────────────────────────────────────────────┘
```

### Step 3 — Constraint-relaxation moves
For each axis where 10× is meaningful, name ONE concrete move:

```text
┌──────────────┬──────────────────────────────┬──────────────────────┬──────────────────────┬──────────────┐
│ Axis          │ The constraint relaxed        │ The move              │ Cost / risk           │ 10× plausible?│
├──────────────┼──────────────────────────────┼──────────────────────┼──────────────────────┼──────────────┤
│ Speed         │ <e.g., sequential per-item>   │ <e.g., batch + vmap>  │ <complexity++; +N hr> │ YES / NO     │
│ ...           │ ...                          │ ...                   │ ...                   │ ...          │
└──────────────┴──────────────────────────────┴──────────────────────┴──────────────────────┴──────────────┘
```

### Step 4 — The honest verdict
End with ONE of:

- `10× achievable on axes [X, Y]. The single highest-leverage move is: <pinpoint>. Expected cost: <hours/$>.`
- `10× NOT achievable on any axis — this solution is near the physical limit. Marginal (2-3×) gains available on: <axis>.`
- `10× possible but requires a categorically different approach — current artifact is the wrong starting point. Suggest: <alternative>.`

## Rules

- **No fairy-tale moves.** "Use GPT-7" or "wait for hardware to improve" is not a move you control.
- **Cost-honest.** A 10× move that takes 2 person-months is fine — name it. A 10× move with no cost label is suspicious.
- **Skip axes where 10× is meaningless.** A daily cron that runs in 30 sec doesn't benefit from 10× speed — write "N/A — already below user-perception floor".

**Distinction from sibling commands**: backward critique = `/brutal` · absences = `/missing` · counter-position = `/steelman` · simplification = `/eli5`. This is uniquely the forward growth axis.
