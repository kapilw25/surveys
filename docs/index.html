Figure · The funnel 13 → 6 → 5 → 3 → 3. As you demand more self-improvement (write once → repair → remember → search → all three), the occupants thin out fast. §3.1.5 — the cell where feedback + memory + search combine into one open-ended loop — is nearly empty, and only ASPIRE ★ was built for it end-to-end. This is the un-surveyed gap the survey claims — no existing survey organizes code-as-policy by self-improvement, or names this cell.

Figure · The "skills" pole. §3.4.1–2 build skills as RL policies (latent / hierarchical — a mature, well-surveyed area, included here as landscape); §3.4.3–4 build them as retrievable code, where ASPIRE's self-improving §3.1 lives. So "weights or skills??" has a second twist: skills themselves come in an RL flavor and a code flavor — and only the code flavor self-improves without gradients.
🌱 Origins (outside the taxonomy): the write → execute → refine → remember loop is borrowed from non-physical software-engineering agents (SWE-agent, OpenHands, Reflexion) and LLM program search (FunSearch, AlphaEvolve) — inspiration only, not physical-AI systems.
§4 · architectures

🏗️ The self-improvement ladder — high-level architecture per sub-section

§3.1.1 Zero-shot synthesisCode-as-Policies · ProgPrompt
📝 instruction + API→ 🧠 LLM→ 🐍 policy code→ 🤖 execute — open loop · no learning from execution
§3.1.2 Closed-loop self-repairREFLECT · DoReMi · Inner Monologue
📝 instruction→ 🧠 LLM→ 🐍 program→ 🤖 execute→ 👀 failure detector ↺ one loop · failure explanation → LLM revises
§3.1.3 Skill-library accumulationVoyager · LRLL · Uni-Skill
🎯 task→ 📚 retrieve ⟵ library→ 🧠 LLM writes skill→ ✅ verify ↺ memory loop · verified skill → library → future tasks
§3.1.4 Evolutionary program searchFunSearch · CaP-X · Code Evolution
🌱 seed programs→ 🏅 execute + score→ ✂️ select top-K→ 🧬 LLM mutates ↺ population loop · mutated programs → next generation
§3.1.5 Full self-improving loop★ ASPIRE · ENPIRE · RoboClaw
🎯 task→ 🤖 actor agent→ 🔬 execution engine (F)→ 📚 skill memory (M)→ 🧬 evolutionary search (S)→ ✅ validated skill ↺ all three loops · traces → actor · skills → library → every future task & embodiment
— the foils (already-surveyed context) —
§3.2 End-to-end VLA (the foil)OpenVLA · RT-2 · π0
🖼️ image+ 🗣️ language→ 🧠 VLM / LLM backbone→ 🎛️ action head→ 🤖 actions — frozen weights · no loop, no code
§3.3 Reward / curriculum synthesisEureka · DrEureka · Text2Reward
⚙️ env code + task→ 🧠 LLM→ 🎁 reward code→ 🎢 RL training→ 📼 policy ↺ reward reflection (training stats) → LLM revises
§3.5 Sim-to-real & cross-embodimentOpen X-Embodiment · CrossFormer
🗂️ 22 embodiments' data→ 📼 single shared policy→ 🤖 deploy on new robot — shared action space · re-collect / fine-tune per robot
— skill-learning: how a repertoire is built (the "skills" pole, §3.4) —
§3.4.1 Unsupervised skill discovery (RL)DIAYN · DADS · METRA
🎲 sample skill z→ 🧠 skill-policy π(a|s,z)→ 🤖 rollout→ 🔎 discriminator q(z|s) ↺ intrinsic reward = skill ↔ state mutual info · no task reward, no code
§3.4.2 Skill-based / hierarchical RLSPiRL · OPAL · SkiMo
🗂️ offline data→ 🧩 learn skill space + prior→ 🎯 high-level policy over skills→ 🎢 downstream RL ↺ skill prior guides exploration · gradient-trained, weights not code
